# Project-Starter Workflow

This project uses a standardized workflow: **idea → spec → tasks → implementation**.

## Entry point

Start with `docs/idea.md`. Fill in What, Why, Rough scope, Constraints, and Open questions.

## Skills

| Skill | When to use |
|-------|--------------|
| `/generate-spec` | After filling `docs/idea.md` – produces `docs/spec.md` |
| `/generate-tasks` | After `docs/spec.md` exists – produces agent-friendly `docs/tasks.md` (granular sub-tasks, acceptance criteria, implementation hints) |
| `/generate-features` | After `docs/tasks.md` exists – produces Gherkin specs in `docs/features/` for key user flows |
| `/start-project` | Run the full flow: idea → spec → tasks (and optionally features) in one go |

## Roles

- **Architect** – Tech stack, structure, patterns (rule: 100-tech-stack)
- **Designer** – UI/UX, design system (rule: 200-ui-ux)
- **QA** – Testing strategy, quality gates (rule: 300-testing)

## Workflow order

1. `docs/idea.md` – Your starting point
2. `docs/spec.md` – Generated by `/generate-spec`
3. `docs/tasks.md` – Generated by `/generate-tasks` (agent-oriented: sub-tasks, File/Contract hints)
4. `docs/features/*.feature` – Optional, generated by `/generate-features` (Gherkin scenarios)
5. Implementation – Follow the task list; use feature specs for acceptance criteria
